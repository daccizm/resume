  - 入社直後にプロダクトのコアな業務領域についての再設計をスタートするプロジェクトが発足し、参画する
  - PO(1),デザイナー(1),開発者(1)の3名のスクラムチームでスタートする
  - ドメイン知識
    - チームで課題仮説や解決仮説を検証するために顧客のインタビューを開始する
    - しかし、顧客調整は依頼から実施までのリードタイムが長いため、スプリント内で得られる学習量の少なさに課題があった
    - そのため、社外だけでなく社内のドメイン領域に詳しい方をドメインエキスパートとしてインタビューし、仮説を検証する活動を追加する
    - その活動によりスプリント内で得られる学習量は確実に上がった手応えを得る
  - ドメインモデル
    - ドメインエキスパートや顧客インタビューにより得たドメイン知識から、miro内で文章、図形などを使いながらドメインを分析する
    - 分析した結果から、ソフトウェアとして実現した時に顧客の課題を解決できる仮説のドメインモデルを設計する
    - ドメインモデル図を作成した後は、オブジェクトの依存先や多重度、属性などに違和感を感じる箇所を検証する
    - 検証方法はレイヤードアーキテクチャで言うアプリケーション層とドメイン層のコードを書き、モデルとして考慮が不足している点を確認する
    - これらの活動を通して感じたことは、顧客にとって価値があるドメインモデルを見つけるには、いかに筋が良いドメインの分析が出来たかにかかっている、と感じる
  - プロトタイプ
    - 仮説のドメインモデルの検証方法としてプロトタイプのアプリケーションを作成する
    - プロトタイプは React, TypeScript でSPAとして構築する
    - バックエンドでのデータ永続化を踏まえて、AWS Amplify を利用したが、プロジェクト終了時まで検証のためにデータを永続化することはなかった
      - Amplify としては UI Component と Hosting のみを利用する結果になった
    - プレゼンテーションはデザイナーがOOUIの有識者であったためドメインモデルをOOUIで表現した設計で進める
  - プロダクトへの適用
    - コアなドメインの業務に関する仮説のモデルが一定検証が完了したため、プロダクトへの適用をスタートする
    - プロダクトの現状のドメインモデルに対して、検証した仮説モデルをFit&Gapし、インクリメントするためのロードマップを作成する
    - 直近のロードマップを実現するための新しいドメインモデルを作成する
    - チームもフロントエンジニア(1), バックエンドエンジニア(1)が新たに参画する
    - 新しいドメインモデルによりプロダクトの影響範囲を調査する
    - 並行して新しいドメインモデルが既存顧客の課題に対してどこまで解決できるかを顧客インタビューで検証する

Engineering Manager
2020年4月 - 2022年5月

  - チーム開発とプロダクトに向き合った開発をするため、自身でコントロールできる範囲を広げやすい自社 SaaS 事業の子会社に出向する
- 開発への取り組み
  - Golang による API 開発
    - 参画当初、顧客からの要望でコミットされていた Golang 製の API の追加開発で人員が足りず、プロダクトの内部を理解するためにも担当を申し出る
    - 初めての Golang であったが、書籍を数冊読み言語について理解し、保守性を考慮してオニオンアーキテクチャで構築する
    - Golang の理解が進んだことも良かったが、実際にプロダクト内部の課題を確認できたことが後のメンバーとの会話で役に立つことになる
  - 要求段階での仮説の検証
    - 新規機能は目的の不確実性が高いため事前に下げておく必要があった
    - 新規機能については要求段階でドメインモデルを定義し、デザイナーと協業し課題が解決できるモデルになっているかを素早く検証する
    - ドメインモデルについてデザイナーと会話しながら、構造設計した UI についてドメインエキスパートと会話する
    - 並行し、ドメインモデルの理解を深めるために、バックエンドの刷新を見据えて Kotlin で TDD により一部実装する
- 開発プロセス改善への取り組み
  - 計画づくりの導入
    - 参画当初はいつ、何がリリースできそうか直近の予定も不透明であり、リリース直後にデグレードが度々発生し、ユーザへの影響も少なくなかったため、安定した品質を定期的にリリースする計画づくりが必要だった
    - まず、デザインディレクターからの提案もあり、イテレーションの期間を1ヶ月に固定する
    - 計画を作るためにも、見積もりのプロセスを追加する。見積もりはサブタスク単位で見積もり、バッファは二乗和平方根法で算出できるよう見積もりシートを作成し運用する
    - 見積もる上で、安定した品質でリリースするためにタスクの影響範囲を調査・分析するプロセスを追加する。その期間で調査した内容を元にタスクのゴールの見直しや見積もりを行う。
  - バグ分析モデリングによる原因分析とプロセス改善
    - リリース後に発生したインシデントはバグ分析のモデリングを実施する
    - モデリングした内容に基づき、再発防止として最も効果が期待できる原因を抽出し、その対策をプロセスに組み込む
      - e.g. 循環的複雑度の計測, 単体テスト定義, 単体テストカバレッジ基準
  - ADR の導入
    - レガシーな技術的負債に対して、背景や意図が不明なことが多く、意図しない設計変更や異なる設計が乱立する状況にあった
    - 新しい設計変更については ADR により意思決定プロセスを明確にし、意思決定の背景や決定事項を残す運用を始める
- 採用への取り組み
  - 各職種のエンジニアの職務定義と質問項目の作成
    - 事業を成長させていくために現在必要なエンジニアの職種とそれらの職種に求められるスキルを整理する
    - 整理したスキルに基づいて要件を作成し、それぞれの質問項目を作成し、エンジニアと採用活動を開始する
    - 回答内容については過去の応募者の回答と比較しながら相対的に評価する。質問内容についても適宜見直しながら運用する
- チームづくりへの取り組み
  - ローコンテキストな習慣づくり
    - 情報が不透明であったために限定合理性が働きやすい環境であった
    - タスクはタスク管理サービスに必ず起票することから始まり、背景やゴール、結果などの記載や同期的コミュニケーションの結果も記載するようチーム全員で取り組む
  - ユニット制の導入
    - 参画当初はマネージャーから個々にタスクが依頼されていたため、属人化や情報の偏りなどがあった
    - 技術領域で区切ったユニットを3~5人で構成し、タスクはユニットが担当するよう変更する
    - ユニットが自律的に活動できるようリードエンジニア、エンジニアの責任範囲を明確にする
  - スクラムの導入
    - 事業成果を最大化させるためにPBIの優先順位の方式を変更した
    - それにより、ユニットの担当領域が必ずしもPBIの上位にないケースがあるため、同時に技術領域単位のユニット制を廃止する必要があった
    - また、PBIの解決仮説の不確実性もコントロールしていく必要があったため、チームは小さくインクリメントしながら仮説を検証する必要があった
    - 以上の背景により、機能横断的なスクラムチームに変更し、プロダクトオーナー、スクラムマスター、開発者の構成でインクリメントしながら仮説検証するチーム体制に変更する
  - 個人ふりかえりの導入
    - 会社の仕組みとして半期に一度自己評価の提出があるが、毎回直前に思い出すところから始まり、具体性に欠ける内容になることも少なくなかった
    - 日々の開発の活動でやったこと、わかったことの学びは多くあるため、それを蓄積する場所を作成し、自身の成長をストックする運用を始める
    - 記載した内容は他のエンジニアからも閲覧でき、フィードバック（感謝と期待）を推奨し、多角的な視点で自身の成長に向き合える環境にする
  - プログラミングの原則理解
    - レガシーな技術的負債に向き合うために、どういうプログラミングが良いのかを共通理解する必要があった
    - [プリンシプル オブ プログラミング3年目までに身につけたい一生役立つ101の原理原則](https://www.amazon.co.jp/%E3%83%97%E3%83%AA%E3%83%B3%E3%82%B7%E3%83%97%E3%83%AB-%E3%82%AA%E3%83%96-%E3%83%97%E3%83%AD%E3%82%B0%E3%83%A9%E3%83%9F%E3%83%B3%E3%82%B03%E5%B9%B4%E7%9B%AE%E3%81%BE%E3%81%A7%E3%81%AB%E8%BA%AB%E3%81%AB%E3%81%A4%E3%81%91%E3%81%9F%E3%81%84%E4%B8%80%E7%94%9F%E5%BD%B9%E7%AB%8B%E3%81%A4101%E3%81%AE%E5%8E%9F%E7%90%86%E5%8E%9F%E5%89%87-%E4%B8%8A%E7%94%B0-%E5%8B%B2/dp/4798046140)はその課題を解消する１つの手段として効果が期待でき、チーム内からの提案により、エンジニアの活動の中に既存コードを元にどういうプリンシプルを適用した方がよかったかというワークショップを開催し、共通の理解ができるよう施策する
  - ホラクラシーを参考にしたロールの導入
    - 自身の退職が決まったが後任のEMは決まっていないため、EMとして持っていたロールをどうするか開発チーム内で検討する
    - 話し合いの結果、チーム内からEMは決めず、全員で分担する意思決定をする
    - EMとして保有していたロールを洗い出し、事業部長と後任が決まるまで保留にできるロールについて合意する
    - 開発組織として最低限必要とする保有していたロールについて、ホラクラシーからロールの目的・目標、ドメイン、責務、ポリシー、ロールリードの考え方を適用する
    - ロールリードについては必要な人数の制約のみとし、チームメンバー自身がwillを持つロールにロールリードとしてアサインし、EM不在だがそれを補う開発チームとして運用を開始する
- セキュリティへの取り組み
  - これまで情報セキュリティに対する課題認識が揃っておらず計画的な対策が遅れることがあった
  - 総務省の「クラウドサービス提供における情報セキュリティ対策ガイドライン」に基づいてセキュリティ対策状況を列挙し、ISMS のスコア付けを参考に評価する
  - プロダクトのセキュリティの状況と優先順位を明確になり、計画的な対策を実施する
- NSM への取り組み
  - 収益以外の指標がなく、プロダクトロードマップの検討のために他の指標を必要としていた
  - 事業の収益性に直結しプロダクトの価値がユーザに届いているか知るための指標として North Star Metric (NSM) を見つけるための活動をする
  - NSM の様々な仮説を立てて、メトリクスをとり、MRR との関連性を調査する
  - その過程でプロダクトのコアの再検討の必要性を感じ、デザインディレクターと検討を開始する
- PBI の優先順位付けへの取り組み
  - 優先順位付けの意思決定理由が明確になっていなかったため、チーム内の理解や効果が不透明であった
  - ICEモデルを検討していたが NSM を定義してからの方が適用しやすいと考え、狩野モデルの５つの品質モデルを参考に評価する
  - しかし、社内の意見のみでの評価は不確実性の度合いが見えず、また優先順位付けの指標にコストも含まれないためROIが不透明な課題があった
  - 最終的に、課題の大きさ、解決量、不確実性、コストを変数とした式で優先順位を算出し決定する

    - 評価については、明瞭であり、公平・公正な評価であることと自身の成長に繋がるフィードバックを得る仕組みづくりを模索する
    - 会社としての職能基準をベースに、チームのエンジニアとして期待されるアセスメント項目を作成し、それをチーム内で相互にフィードバックする。そのフィードバック内容を数値化し、グレード別に点数を設定し、評価する仕組みを構築する
    - メンバーへのフィードバックはアセスメントのフィードバック内容と1on1の内容を踏まえることにより、それぞれの視座・視点・視野から多角的にフィードバックするよう努める
- 目標への取り組み

    - ミッションは OKR のチームの Objective にも関連させることにより、チーム内の全ての活動がミッションに繋がることを前提とした設計を行う
    - 受託開発が事業のメインだったこともあり、チームとして関わるメンバーが固定されやすい状況だったため、ミッションを達成するために必要なことをそれぞれのメンバーが考え、それをサブミッションとして共感できるメンバー同士をユニットとして構成し活動する

  - チーム全体の理解度の向上のために「ドメイン駆動設計モデリング/実践ガイド」等の著者でもある [松岡幸一郎さん(@little_hand_s)](https://little-hands.hatenablog.com/) を外部講師としてお招きし、DDD のワークショップを開催する
- リスクアセスメントへの取り組み
  - 各案件の未来への不確実性についてそれぞれのプロジェクトが意識し、管理できるようにチーム内でリスクアセスメントの棚卸し会を実施する。そこでは、未来に対する懸念や不安について立ち止まって考え、相談する場として実施する。
