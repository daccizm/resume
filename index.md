#### プロダクトの再設計（ドメイン理解）
入社直後にプロダクトのコアな業務領域についての再設計をスタートするプロジェクトが発足し、PO(1),デザイナー(1),開発者(1)の3名のスクラムチームで開発者として参画した。

まず、業務理解を深めるために社内外のドメインエキスパートに業務内容をヒヤリングし、得たドメイン知識から miro にてユースケース図、状態遷移表、文章、図形などを使いながらドメインを整理した。

分析した結果からソフトウェアとして実現した時に課題を解決できる仮説のドメインモデルを定義し、その検証としてプロトタイプのアプリケーションを作成した。

プロトタイプは AWS Amplify, React, TypeScript で SPA として構築した。データ永続化を踏まえて、AWS Amplify を利用したが、Discovery 段階では静的なデータでのプロトタイプでも十分に学びが得られることがわかり、Amplify は UI Component と Hosting のみを利用した。

結果、積み上げた検証結果から理想とするドメインモデルを図示したものを成果物として既存のプロダクトへの適用に移行した。

ドメインモデル図は、集約、値オブジェクト、エンティティ、各モデルとの多重度や参照先や例としてのドメインオブジェクトを記載した。

#### プロダクトの再設計（既存プロダクトへの適用）
前のプロジェクトでプロダクトのコアな業務領域について仮説となるドメインモデルを作成した後、既存のプロダクトにそのドメインモデルを適用するプロジェクトが開始した。

仮説のドメインモデルをそのままプロダクトに適用すると既存顧客への影響が大きく業務に支障をきたす懸念があったため、顧客の業務にマイナスの影響を起こさず理想と現実の業務モデルのギャップを埋めたドメインモデルを探索する必要があった。

そのため、適用するドメインモデルを探索する Discovery 担当と、検証した結果をプロダクトに適用する Delivery 担当に分割し、プロジェクトを進める。

プロジェクトは PO(1), デザイナー(1), 開発者(3) の5名のスクラムチームで開発者として参画した。Discovery 担当は私とデザイナー(1)、Delivery 担当は開発者(2) の構成になる。

まず、プロダクトの現状の業務モデルと検証した業務モデルを Fit & Gap し、インクリメントするためのロードマップを作成した。その時、既存のプロダクトへの影響が少なく課題の解決量が大きい順序で作成する。

プロダクトに適用する業務モデルを再定義し、それを元にデザイナーが miro でワイヤーフレームを作成し、社内外のユーザーにヒアリングする。業務モデルの有用性がある程度確信できた後は、ドメインモデルの振る舞いや構造に落ち度がないかを検証するためにオニオンアーキテクチャでいうアプリケーション・サービス層とドメインモデル層のコードを書く。

適用するドメインモデルの全体像が見え始めたころ、具体的なリリースラインを決める必要を感じたため、ユーザーストーリーマッピングを作成した。プロダクトに適用する業務領域の範囲内での主要なユーザーの行動とそれにまつわるユーザーストーリーを作成し、どのユーザーストーリーまで対応すればリリース可能かPOと会話し策定した。

Delivery 担当へのインプットとしては以下を記載したユーザーストーリーと検証したドメインモデル、ワイヤーフレーム、受け入れ条件になる。

- 顧客がどういう業務をしているか
- そこの課題は何があるか
- その課題に対して既存のプロダクトは何を解決しているか、そして、何が解決できていないか
- それを今回どのように解決するか

結果、現在顧客へのテスト提供の打診を開始しその反応は好調であり、競合他社製品からの乗り換え相談に必須要件の中に提供予定の機能が含まれていたことから業務領域を理解したモデルが作成できていることに確信が得られている。

#### 新規事業

会社のビジョン達成のために toC 向けの事業を展開する話があり、かねてより新規事業に携わりたかったため手を挙げ参画した。チームは、PO(1), PjM(1), デザイナー(1), 開発者(1) であり開発者として参画した。

まずは、ペルソナイメージとその課題とソリューションの方向性を明確にする CPF のフェーズを取り組んだ。

チームメンバーは兼任が多く、同期的に会話して物事を決める時間が圧倒的に少なかったため、学習のサイクルを早く回すためにも意思決定スピードと検証すべき論点を明確にして検証していく必要があった。

そのため、ジェベリンボードを使いアクターと課題とソリューションと検証すべき前提条件を１〜２時間のミーティングで決める必要があり、ブレスト後は基本的に PO がどんどん意思決定していくスタイルにした。

私はファシリテートと、決まったことをジェベリンボードで整理し、インタビュースクリプトやインタビュー実施を担当した。

まだ進行中ではあるが、この進め方によりプロジェクト開始から２週間で3回の仮説更新と9回のインタビューが実施できており、検証速度は概ね順調である。

  - 計画づくりの導入
    - 参画当初はいつ、何がリリースできそうか直近の予定も不透明であり、リリース直後にデグレードが度々発生し、ユーザへの影響も少なくなかったため、安定した品質を定期的にリリースする計画づくりが必要だった
    - まず、デザインディレクターからの提案もあり、イテレーションの期間を1ヶ月に固定する
    - 計画を作るためにも、見積もりのプロセスを追加する。見積もりはサブタスク単位で見積もり、バッファは二乗和平方根法で算出できるよう見積もりシートを作成し運用する
    - 見積もる上で、安定した品質でリリースするためにタスクの影響範囲を調査・分析するプロセスを追加する。その期間で調査した内容を元にタスクのゴールの見直しや見積もりを行う。
  - バグ分析モデリングによる原因分析とプロセス改善
    - リリース後に発生したインシデントはバグ分析のモデリングを実施する
    - モデリングした内容に基づき、再発防止として最も効果が期待できる原因を抽出し、その対策をプロセスに組み込む
      - e.g. 循環的複雑度の計測, 単体テスト定義, 単体テストカバレッジ基準
  - ADR の導入
    - レガシーな技術的負債に対して、背景や意図が不明なことが多く、意図しない設計変更や異なる設計が乱立する状況にあった
    - 新しい設計変更については ADR により意思決定プロセスを明確にし、意思決定の背景や決定事項を残す運用を始める

  - 各職種のエンジニアの職務定義と質問項目の作成
    - 事業を成長させていくために現在必要なエンジニアの職種とそれらの職種に求められるスキルを整理する
    - 整理したスキルに基づいて要件を作成し、それぞれの質問項目を作成し、エンジニアと採用活動を開始する
    - 回答内容については過去の応募者の回答と比較しながら相対的に評価する。質問内容についても適宜見直しながら運用する
- チームづくりへの取り組み
  - ローコンテキストな習慣づくり
    - 情報が不透明であったために限定合理性が働きやすい環境であった
    - タスクはタスク管理サービスに必ず起票することから始まり、背景やゴール、結果などの記載や同期的コミュニケーションの結果も記載するようチーム全員で取り組む

    - 技術領域で区切ったユニットを3~5人で構成し、タスクはユニットが担当するよう変更する
    - ユニットが自律的に活動できるようリードエンジニア、エンジニアの責任範囲を明確にする

  - ホラクラシーを参考にしたロールの導入
    - 自身の退職が決まったが後任のEMは決まっていないため、EMとして持っていたロールをどうするか開発チーム内で検討する
    - 話し合いの結果、チーム内からEMは決めず、全員で分担する意思決定をする
    - EMとして保有していたロールを洗い出し、事業部長と後任が決まるまで保留にできるロールについて合意する
    - 開発組織として最低限必要とする保有していたロールについて、ホラクラシーからロールの目的・目標、ドメイン、責務、ポリシー、ロールリードの考え方を適用する
    - ロールリードについては必要な人数の制約のみとし、チームメンバー自身がwillを持つロールにロールリードとしてアサインし、EM不在だがそれを補う開発チームとして運用を開始する

  - これまで情報セキュリティに対する課題認識が揃っておらず計画的な対策が遅れることがあった

- NSM への取り組み
  - 収益以外の指標がなく、プロダクトロードマップの検討のために他の指標を必要としていた

  - NSM の様々な仮説を立てて、メトリクスをとり、MRR との関連性を調査する
  - その過程でプロダクトのコアの再検討の必要性を感じ、デザインディレクターと検討を開始する
- PBI の優先順位付けへの取り組み
  - 優先順位付けの意思決定理由が明確になっていなかったため、チーム内の理解や効果が不透明であった
  - ICEモデルを検討していたが NSM を定義してからの方が適用しやすいと考え、狩野モデルの５つの品質モデルを参考に評価する
  - しかし、社内の意見のみでの評価は不確実性の度合いが見えず、また優先順位付けの指標にコストも含まれないためROIが不透明な課題があった
  - 最終的に、課題の大きさ、解決量、不確実性、コストを変数とした式で優先順位を算出し決定する
